Transportation Routing System

It's always been fascinating how routing algorithms take vast maps and instantly generate the quickest way to traverse them. I don't know how to do complex path-finding algorithms; however, I know how to make object-based programs. Thus, it'd be interesting to build a simulator customized to a more unique scenario with more parameters. In a way, it's an expansion and modification onto the elevator simulator project.

When the transportation simulator runs, the user must input a list of cargo that needs to be delivered through any of the given routes (which they also provide). Many different vehicles can be chosen, each with its own benefits and drawbacks. For example: A semi-truck can't go through cities and has a road tax, but it can carry a lot of cargo. A bicycle is slow and has to take breaks often, but it has a low operating cost. Given these constraints, the program then uses different routers to assign the cargo to specific combinations of cars and routes (e.g., the cheapest option or the quickest option). It then repeatedly simulates the motion of these vehicles until they deliver all of the cargo.

We decided to continue with Java as we were all familiar with it. We initially started with a different program, one that involved socket communication. It was a fascinating idea; however, some of us (me personally) lacked experience on it, and it didn't prove easy to test as it involved running multiple instances / servers for communications between them. There was no easy centralized way to run the entire program at once. Thus, we pivoted to something we were interested in as well and more familiar with, which was another simulator project. It was a bit of a disappointment and left us with less time to craft up a complex project.

I drove a lot of the over-arching template for the project. I wanted to change it from sockets as I was unable to help with that. Eli and Thomas had both taken networking, but I haven't. Thus, I guided the transition to a different project and laid out the general plan. I created the requirements for what parts of the code should do and coded up some of it. We all collaborated to produce the project and make it what it is. It was challenging to brainstorm exciting ideas for the project. I wanted something that was able to be tested well (e.g., something without a GUI, as that is quite difficult to test) while still being interactive for the user. This elimated many different game ideas as they'd have to be console-based.
